1 Introduction {#mainpage}
2 ============
3 
4 [Cynara](https://wiki.tizen.org/wiki/Security:Cynara) is fast, simple and safe policy-checker service.
5 Functions of Cynara are:
6 * checking access for certain privileges
7 * holding policies
8 * simple, single function API - for checking policies
9 * thin client library - to make access control even more simple
10 * ability to use external agent
11 (in case of policies that can't be full processed in cynara and plugins)
12 
13 # API
14 Cynara's API is split into two parts:
15 * client API for privilege checking API - easily accessed with libCynara by anyone
16 * admin API for policy managing API - accessed only by trusted processes with libCynaraAdmin.
17 
18 Cynara service uses separate Unix Domain Sockets for handling each type of API.
19 
20 ## Client
21 Both synchronous and asynchronous client API for cynara is available. It is provided by
22 libcynara-client and libcynara-clienty-async libraries.
23 
24 [Details of client synchronous API](@ref cynara-client.h)
25 
26 [Details of client asynchronous API](@ref cynara-client-async.h)
27 
28 To enhance client performance a simple LRU Cache was implemented for both synchronous and
29 asynchronous client API. Cache is available since cynara 0.2.0 version.
30 
31 [Details of cache](https://wiki.tizen.org/wiki/Security:Cynara:API:client-cache)
32 
33 On Tizen 3.0 application context - information that identify application and user that runs it
34 are defined by SMACK label and UID. There are helpers libraries for different IPC. They will make
35 obtaining these information easier. They are described below in credentials section.
36 
37 There might be a problem, if IPC between application and service do not provide UID and SMACK label
38 of application. The only way to obtain application context is probably reading it from /proc/PID
39 (service needs RX to application SMACK label for that).
40 
41 Direct ways of getting these information is described [here][2], but users are strongly recommended
42 to use credential helpers libraries.
43 
44 ## Admin
45 Admin API is the only way to change policy kept in cynara. Its usage needs root privileges on Tizen
46 . This restriction guarantees that no unprivileged process can play with cynara policy.
47 On Tizen 3.0 SecurityManager is a service dedicated to make changes to cynara and probably should
48 be the only user of cynara's admin interface. Processes that would like to change cynara policy
49 (e.g. installers) should use libsecurity-manager-client library instead of direct cynara calls.
50 
51 [Details of admin API](@ref cynara-admin.h)

53 ## Plugins
54 ### Description
55 Plugin API enables creating independent plugins for Cynara extensions. Plugins are shared libraries
56 dynamically loaded into Cynara. They provide entry points defined by API to create external plugin
57 entities. Plugin API is based on class interfaces, which define methods needed for both client-
58 and service-side plugins.
59 
60 ### Client plugin
61 Client plugins are responsible for interpreting external policy types, which can be delivered
62 to client as cynara response, and translate them to [CYNARA_API_ACCESS_ALLOWED][3] or
63 [CYNARA_API_ACCESS_DENIED][4]. Plugins are only allowed to interpret policy types declared by
64 overridden [1] method. Client plugins usually come together with service plugin (for e.g. askuser
65 plugin – available in platform/core/security/askuser tizen.org repository).
66 
67 ### Service plugin
68 Service plugins are responsible for interpreting external policy types. Communication with Cynara's
69 agant extensions could be required for this to work. Plugins are only allowed to interpret policy
70 types declared by overridden [1] method. Service plugins usually come together with Cynara agent
71 extension (for e.g. askuser agent – available in platform/core/security/askuser tizen.org
72 repository).
73 
74 [Details of plugin API](@ref cynara-plugin.h)

79 ## Credentials
80 The helper APIs can be used to create:
81 * client identification strings
82 * user identification strings
83 
84 Identification strings are necessary to specify Cynara policies. However, it is important to notice
85 that usage of strings generated by these helper APIs is not obligatory – a different consistent way
86 of client and user process identification strings might be introduced.
87 
88 The client identification string might be generated based on the client process PID or SMACK label.
89 The user identification string might be generated based on the client process UID (user id) or
90 SMACK label. The default methods for a system might be obtained by calling
91 cynara_creds_get_default_client_method or cynara_creds_get_default_user_method respectively.
92 
93 Currently, there are two sets of functions which generate the identification strings.
94 The one is based on DBUS connection, the other is based on Unix socket file descriptors.
95 The connections and socket descriptors are used to obtain the PID, SMACK or UID associated
96 with the client or user process.
97 
98 In addition to the functions generating the identification strings, there are functions returning
99 the PID of the process on the other side of socket/dbus connection. The PID might be used to
100 generate different identification strings.
101 
102 Details of credentials helper libraries:
103 [commons](@ref cynara-creds-commons.h),
104 [dbus](@ref cynara-creds-dbus.h),
105 [gdbus](@ref cynara-creds-gdbus.h),
106 [socket](@ref cynara-creds-socket.h).
107 
108 ## Session
109 This helper API can be used to create a client session string identifiers based on PID and time of
110 creation a client process. Session string identifiers are used in in cynara_check() and
111 cynara_async_create_request() functions defined in client libraries. It is important to notice that
112 usage of strings generated by session library APIs is not obligatory – a different consistent way
113 of identifiers generation might be introduced.
114 
115 [Details of session helper library](@ref cynara-session.h)
116 
117 ## Agent
118 Cynara plugins may use external applications (agents) in order to perform some action on privilege
119 check request. [Agent API](@ref cynara-agent.h) allows communication between plugins and agents.
